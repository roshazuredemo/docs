AKS Platform — Infrastructure Organisation

📅 v1.0 · Draft 🔒 Internal — Restricted 🌏 Azure New Zealand North ☁️ 2 Tenants · 3 Subscriptions · 3 Clusters

Overview

This document describes how the AKS platform is organised from the top of the Azure structure down to individual Kubernetes namespaces. The intent is to give any team member — engineer or architect — a clear mental model of where things live and why.

The platform is spread across two Entra ID tenants and three Azure subscriptions. Each subscription maps to a stage of the delivery pipeline — Dev, Test, and Prod — and contains a dedicated AKS cluster, a shared resource group for common services, and environment-specific resource groups for databases and file storage. Inside each cluster, workloads are isolated by Kubernetes namespace, one per environment.

🏢 2 Entra ID Tenants
Non-Prod and Prod tenants provide a hard identity boundary. Dev and Test subscriptions sit in Non-Prod. Production sits in its own tenant.
📦 3 Subscriptions
Dev, Test, and Prod — one per delivery stage. Each subscription is the Landing Zone unit and comes with a pre-allocated VNet and 3 subnets.
⚙️ 3 AKS Clusters
One cluster per subscription. Environments are separated by Kubernetes namespace within each cluster, not by separate clusters.

Tenant Strategy

Two Entra ID tenants provide a hard boundary between non-production and production identities. All SPNs, Managed Identities, app registrations, and Entra groups in the Non-Prod tenant are completely separate from those in Prod. A credential leak in Non-Prod cannot be used against the production environment.

Non-Production Tenant
Dev Subscription
ACR Key Vault Dev Cluster rg-app-sandbox rg-app-dev
Test Subscription
ACR Key Vault Test Cluster rg-app-intgtest rg-app-systest rg-app-uat rg-app-perftest
Production Tenant
Prod Subscription
ACR Key Vault Prod Cluster rg-app-prod
Why two tenants?
A single tenant would mean a compromised Non-Prod SPN could potentially access Prod Key Vaults or ACR. Separate tenants make that impossible — there are no cross-tenant trust relationships configured.

Subscription Organisation

Each subscription follows the same resource group structure. This consistency makes pipelines reusable and makes it easy to reason about what is in each subscription without looking it up.

Resource Group Structure (same pattern in all three subscriptions)

Resource GroupWhat lives hereShared or per-env?
rg-network-{sub}VNet, subnets, NSGs, route tables, Private DNS zonesShared — one per subscription
rg-common-{sub}ACR, Key Vault, Private Endpoints for both, Managed IdentitiesShared — one per subscription
rg-aks-{sub}AKS cluster resource, node pools, cluster Managed IdentityShared — one per subscription
rg-app-{env}PostgreSQL VM, Storage Account + SFTP file share for that environmentPer environment — one RG per env

Common Resources (rg-common-{sub})

ACR and Key Vault sit at the subscription level and are shared across all environments within that subscription. This avoids duplicating container image storage per environment. The Key Vault holds all secrets for all environments in that subscription, separated by a naming prefix per environment (e.g. sandbox-db-connstr, dev-db-connstr).

ResourceDev SubTest SubProd SubNotes
ACRdevacr001testacr001prodacr001Standard SKU for Dev/Test, Premium for Prod
Key Vaultdev-akv-001test-akv-001prod-akv-001Secrets namespaced by env prefix
ACR accessAKS Managed IdentityAKS Managed IdentityAKS Managed IdentityAcrPull — manually assigned by Cloud Team
AKV accessWorkload Identity per envWorkload Identity per envWorkload Identity per envMounted to pods via CSI Driver
Image Promotion
Images are built once and pushed to the Dev subscription ACR. After validation they are promoted to Test ACR, then to Prod ACR — never rebuilt. The same binary that passed testing is what runs in production.

Network Design

Each subscription gets one VNet pre-allocated by the Azure Landing Zone team. The VNet is connected to the Hub VNet via peering, which routes traffic to on-premises via ExpressRoute. All Azure resources are provisioned as private — no public IPs, no public endpoints.

VNet & Subnet Layout (same in all three subscriptions)

SubnetPurposeExample CIDR
snet-vm Virtual machines — PostgreSQL database VMs, any other IaaS VMs. These are in the same VNet as AKS so pods reach them directly by private IP without a Private Endpoint. 10.x.0.0/24
snet-services Private Endpoints for PaaS services — ACR, Key Vault, Storage Accounts (SFTP). All PaaS resources project into this subnet as a private IP so they are never reachable over the public internet. 10.x.1.0/24
snet-aks AKS node pools. All cluster nodes (system, platform, application) run in this subnet. Pod-to-pod and pod-to-VM traffic stays within the VNet. 10.x.2.0/22

Key Connectivity Rules

FromToHow
On-premises / CloudBees agentsAKS API server, ACR, Key VaultExpressRoute → Hub VNet → VNet peering → Spoke VNet. All port 443. Requires explicit firewall rule approval.
AKS podsPostgreSQL VMsDirect within VNet — pod on snet-aks reaches VM on snet-vm by private IP. Port 5432. No Private Endpoint needed.
AKS podsKey Vault, ACR, StoragePrivate Endpoint in snet-services. Resolved via Private DNS. No internet traversal.
On-premises physical clientsSFTP StorageExpressRoute → Hub → Private Endpoint in snet-services. Port 22.
AKS nodes (all)On-prem observability (ELK, Dynatrace)ExpressRoute return path. Requires firewall rules for log/metrics forwarding ports.
Default-deny posture
The Hub firewall blocks everything by default. Every connectivity path above requires an explicit firewall rule submitted to the Cloud Team before it will work. Plan and submit all rules as a batch before beginning cluster provisioning.

AKS Cluster Design

Each subscription has one AKS private cluster. The clusters differ in topology depending on the environments they serve — Dev is lightweight with a single AZ, Test is single-AZ but larger, Prod is multi-AZ for high availability.

ClusterSubscriptionAZEnvironments (namespaces)System poolPlatform poolApp pool
aks-dev Dev Single AZ sandbox, dev 2 nodes 2 nodes 2–6 nodes (KEDA)
aks-test Test Single AZ intgtest, systest, uat, perftest 2 nodes 2 nodes 2–10 nodes (KEDA)
aks-prod Prod Multi-AZ (3) prod 3 nodes (1/AZ) 3 nodes (1/AZ) 3–30 nodes (KEDA)

Three Node Pools — Why

Separating workloads across node pools means cluster-level components cannot be evicted by application load, and application workloads cannot starve platform tooling. Each pool has its own scaling, instance type, and taint/toleration configuration.

System
pool
Runs core Kubernetes components only. Tainted so application and platform pods cannot schedule here. Not auto-scaled — always on.
kube-system CoreDNS · kube-proxy · cloud-node-manager
Platform
pool
Runs all shared operator workloads. Tainted so application pods cannot schedule here unless explicitly tolerated. Fixed size — not event-driven autoscaled.
istio-system monitoring logging cert-manager rabbitmq-system opentelemetry jaeger keda secrets-store-csi kyverno
Application
pool
Runs all application workloads. KEDA-autoscaled based on RabbitMQ queue depth. Pods are isolated per environment by namespace, NetworkPolicy, and ResourceQuota.
app-sandbox app-dev app-intgtest app-systest app-uat app-perftest app-prod

Namespace & Node Pool Layout

Every namespace is deployed to a specific node pool using node affinity and taints. This is enforced by Kyverno at admission time — a pod that does not declare the correct affinity is rejected.

Fluent Bit note
Fluent Bit is deployed as a DaemonSet to the logging namespace but runs a pod on every node across all three pools. This ensures no logs are missed regardless of which node pool a workload runs on.

Environment Strategy

Environments are separated by Kubernetes namespace within each cluster. Each namespace is fully isolated — it has its own NetworkPolicy (default-deny, created automatically by Kyverno), its own ResourceQuota, its own RBAC bindings, and its own Workload Identity connecting to environment-specific Azure resources.

ClusterEnvironmentNamespacePurposeDeploy trigger
aks-devSandboxapp-sandboxFree experimentation, spike testing, quick feature demos. Lightweight — single DB replica, minimal SFTP data.Manual or PR open
Devapp-devActive feature development. Connected to dev databases and dev file share. Deployed from feature branches.Auto on feature/* PR
aks-testIntgTestapp-intgtestIntegration testing — verifies service-to-service contracts and downstream dependencies.Auto on develop merge
SysTestapp-systestFull system testing with realistic data volumes. Closest to production configuration in non-prod.Auto on develop merge
UATapp-uatUser acceptance testing — business stakeholders validate before release sign-off.Manual on release/* branch
PerfTestapp-perftestPerformance and load testing. Namespace can scale aggressively — isolated quota to prevent cluster impact.Manual on demand
aks-prodProdapp-prodProduction workload. Multi-AZ cluster. KEDA autoscaling. Manual approval gate required.Manual approval on main

Namespace Bootstrap — What the Pipeline Creates

When a new environment namespace is created the pipeline applies these in order. Nothing is manually configured after this point.

#WhatHow
1Namespace with standard labelskubectl apply — labels include environment, team, cost-centre
2Default-deny NetworkPolicyAuto-generated by Kyverno Generate policy on namespace creation
3ResourceQuota and LimitRangeApplied from cluster-baseline Helm chart per environment tier
4Kubernetes ServiceAccount for Workload IdentityLinked to an Azure Managed Identity via Federated Credential
5SecretProviderClassDefines which Key Vault secrets mount into pods in this namespace
6RBAC RoleBindingsSPN for pipeline deploy + Entra groups for human read access
7Application Helm releaseDeployed by the environment-specific pipeline SPN

Identity & RBAC

There are three types of identity on this platform — pipeline SPNs for automation, Managed Identities for pod-to-Azure connectivity, and Entra ID groups for human access. They are kept strictly separate and scoped to the minimum required permissions.

Pipeline SPNs

Each SPN is an App Registration in the relevant tenant. One SPN per automation task — no sharing.

SPNTenantScopeRoleDoes
spn-infra-{sub}Per tenantrg-common, rg-app-* in that subContributorProvisions ACR, Key Vault, Storage Accounts, PostgreSQL VMs
spn-cluster-{sub}Per tenantrg-aks, rg-network in that subContributor + Network ContributorProvisions AKS cluster, node pools, subnet delegation
spn-image-push-{sub}Per tenantACR resource onlyAcrPushPushes built images to subscription ACR
spn-image-promoteNon-Prod → ProdSource ACR (read) + Dest ACR (write)AcrPull + AcrPushPromotes approved images from lower to higher subscription ACR
spn-k8s-baselinePer tenantK8s: cluster-admin (all clusters)cluster-adminInstalls system operators via Helm. Pipeline only — never interactive.
spn-app-{env}Per tenantK8s: namespace-deployer in that env namespacenamespace-deployerDeploys application Helm releases to the specific env namespace

Managed Identities (Pod-to-Azure)

Pods never use SPNs directly. Workload Identity federates a Kubernetes ServiceAccount to an Azure Managed Identity. This means no secrets in pods — the identity token is projected automatically and is short-lived.

Role assignment constraint
The Landing Zone policy prevents any SPN from assigning Azure roles. The AcrPull assignment from the AKS cluster Managed Identity to ACR must be raised as a Cloud Team request after each cluster is provisioned. Build this into the cluster provisioning runbook as a mandatory step.
