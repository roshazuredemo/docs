00

Why this session exists

We have been asked to land the vendor-supplied UP Framework v5.1.3 onto AKS in our existing Spoke VNet, with all access private over ExpressRoute, Entra ID for identity, Vault for secrets and CloudBees for delivery. Most of those constraints are non-negotiable. What is negotiable — and what this session decides — is the shape of the cluster estate: how many clusters, how environments are partitioned, which tools we adopt, which we defer, who owns what after Day 0, and which questions we send to Microsoft, to Cyber, and to the ACI vendor before design freeze.

The brief is deliberately short. Each topic carries two to four options, an annotated recommendation, and the open questions we should put in the parking lot. The goal is to leave the room with about a dozen locked decisions, and a clear list of items waiting on someone outside the room.

02

Cluster topology

How many clusters do we run, and what does each cluster contain? This is an irreversible-ish decision — collapsing two clusters into one is feasible but expensive; splitting one in two later is brutal. We commit upfront.

OPT-A reject

Single shared cluster

One cluster hosts Dev, NP and Prod environments via namespaces.

  • Lowest cost & one upgrade window
  • Blast radius covers Prod
  • Cyber will not approve — Prod and Dev sharing control plane
OPT-B recommended

Three clusters: Dev · NP · Prod

Independent clusters per environment tier. NP holds SIT/UAT/PERF as namespaces. Prod is single-tenant.

  • Clean blast radius, independent upgrade cadences
  • Distinct change-windows for Prod
  • Aligns with our existing Maturity Model (Part 01)
  • Three control planes to operate
OPT-C defer

Five clusters: Dev / SIT / UAT / PERF / Prod

One cluster per logical environment.

  • Strongest isolation — no shared control plane in NP
  • 5× cost for control planes & node minimums
  • Operational overhead does not justify isolation gain at this scale
OPT-D defer

Cluster-per-tenant (UP-FW, future tenants)

Provision a new cluster every time a new line-of-business product lands.

  • Hard isolation per product
  • Premature — we have one tenant (UP-FW) on Day 1
  • Cluster sprawl without a platform team mature enough to operate it
Working recommendation — to confirm in slot 02

OPT-B (three clusters). Naming: aks-dev-up, aks-np-up, aks-prod-up. Dev and NP in primary region only; Prod resilient across 3 AZs in primary, with a DR posture deferred to Part 01.

03

Environment separation inside a cluster

The Non-Prod cluster hosts SIT, UAT and PERF. How do we keep them apart so a noisy PERF run does not starve UAT, and so a misconfigured SIT chart cannot reach UAT data?

OPT-A recommended

Namespace-per-env tenancy

SIT, UAT, PERF are namespaces. Isolation = NetworkPolicy default-deny + Istio mTLS + AuthorizationPolicy + ResourceQuota + LimitRange + PSA.

  • Native, supported, well-documented
  • Fits the four-layer isolation model in Part 03
  • Soft tenancy — discipline-dependent
OPT-B conditional

Namespace + dedicated node pool for PERF

PERF lane gets its own taint/tolerance node pool to prevent noisy-neighbour impact during load tests.

  • Removes PERF as a noisy-neighbour risk
  • Cheap incremental change on top of OPT-A
  • Idle PERF pool burns money outside test windows — needs auto-scale to zero
OPT-C defer

vClusters / Capsule / Loft

Each env gets a virtual control plane on top of the host cluster.

  • Stronger API-level isolation per tenant
  • Net-new operating model — Cyber, SRE, support all need uplift
  • Not justified for one tenant on Day 1
Working recommendation — to confirm in slot 03

OPT-A as baseline, with OPT-B activated for PERF only if (and when) first capacity tests reveal noisy-neighbour interference on UAT. Dedicated node pool added as a Day 1.5 item, not Day 0.

05

Identity & RBAC

Three identity surfaces: human access to the API server, pipeline access, and pod-to-Azure access. We pick the integration mode for each. Detailed persona & group hierarchy is owned by Part 02 — this slot just locks the integration mode.

RBAC-A recommended

Human access

AKS-managed Entra ID + Azure RBAC for Kubernetes. Local accounts disabled. Cluster-admin via PIM only.

  • One auth source, audit trail in Entra
  • No kubeconfig shipping
  • Aligns with bank-wide Conditional Access posture
RBAC-B recommended

Pipeline identity

SPN with OIDC federation from CloudBees. No static client secret anywhere. Two SPNs per env (Terraform applier & Helm deployer).

  • Zero static credentials
  • Subject pinned to CloudBees folder + pipeline
  • Requires CloudBees OIDC support — confirm version
RBAC-C recommended

Pod-to-Azure

Workload Identity (OIDC) — ServiceAccount → Federated Credential → Azure permissions. No managed identity hijacking, no IMDS shim.

  • Per-workload, namespace-scoped
  • Vault uses the same federation for secret retrieval
Working recommendations — to confirm in slot 05
  • · Local accounts disabled on every cluster (no break-glass kubeconfig).
  • · Cluster-admin PIM-eligible only — no standing privilege.
  • · Six personas (Platform Admin, Cluster Admin, App Deployer, SRE, Auditor, Break-glass) — full hierarchy in Part 02.
  • · Break-glass = single Entra account per cluster, PIM-activated, MFA-enforced, monitored.

07

Pipeline & deployment identity

CloudBees is the only pipeline runner. There are two distinct delivery flows: Terraform that provisions Azure resources (cluster, ACR, KV, peerings), and Helm that deploys workloads into clusters. They must use different identities.

PIPE-A reject

Single SPN per env, secret stored in CloudBees

  • Simple to configure
  • Static secret — rotation burden, theft risk
  • One identity covers infra and app — over-privileged
PIPE-B recommended

Two SPNs per env, OIDC federation from CloudBees

  • No static secret — federation is the credential
  • Subject pinned to a folder + pipeline name
  • Splits Terraform-applier (broad infra) from Helm-deployer (namespace-only)
  • Requires CloudBees OIDC plugin — confirm

The single rule for downstream connectivity

Every outbound connection from a pod is brokered by an identity (Workload Identity → Vault → service auth). No long-lived credential is ever written to a pod's filesystem or environment. If a downstream cannot fit that pattern, it goes to the parking lot.

09

Policy, CIS & cyber gates

Three layers of policy apply to this cluster. Bank-level Azure Policy assignments control what we can provision in the subscription. CIS benchmarks define a baseline cyber will measure us against. K8s admission controllers enforce policy at deploy time inside the cluster. We need a clear picture of all three before cyber sign-off.

Layer 1

Bank Azure Policy initiative

Inherited from Landing Zone at Management Group scope. We do not opt out — we add cluster-specific deny rules on top.

  • · Deny public IP on AKS
  • · Deny non-private API server
  • · Deny ACR public access
  • · Require diagnostic settings
  • · Require tags & resource locks
  • · Defender for Cloud — Containers plan ON
Layer 2

CIS benchmarks — three of them

Three distinct benchmarks apply; we should not conflate them.

  • CIS Azure AKS Benchmark — control plane / cluster config
  • CIS Kubernetes Benchmark — workload manifests, RBAC, NetworkPolicy
  • CIS Docker Benchmark — base image & build pipeline

Defender for Cloud's regulatory compliance dashboard maps the AKS one. The others we measure ourselves (kube-bench, Trivy).

Layer 3

In-cluster admission

Controls applied at deploy time, before a pod ever runs.

  • · Pod Security Admission — baseline (NP), restricted (Prod)
  • · Kyverno (recommended) policies — image registry pin, label requirement, mTLS sidecar requirement
  • · Image signing — cosign / Ratify — Prod only initially, warn-mode in NP
  • · Trivy operator — runtime scan, fail-build threshold
